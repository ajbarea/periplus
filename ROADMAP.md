# Periplus Roadmap

Periplus is a fork-it-yourself, offline-first travel-itinerary PWA. **Edit mode** now lets you change every field on the page with zero code — and add, remove, and reorder cards, list items, *and* whole weeks / checklist groups (directory cards, week plans, checklist items, timeline rows, whole weeks, whole groups) — saved on-device, exportable as JSON; the `TRIP` block in `app/index.html` remains the source for forking. The north star: full no-code authoring — add, remove, and reorder the top-level page sections too.

## Now (shipped)
- **Single source of truth** — every section and card renders from the `TRIP` object: directories, calendar, timelines, intel, checklists, venue, contacts. Edit one block, redeploy, it's your trip.
- **Tap-to-navigate** (Google/Apple Maps) and **add-to-calendar** (Google/Apple) on every relevant item.
- Installable, offline-first, prints clean.

## Next: no-code editing — remaining structure
Stages 1 (field editing), 2 (card sections), 3 (list & thin sections), and 4 (whole weeks & groups) shipped (see Done). Remaining:

- **Sections** — add, delete, reorder, rename, and pick a type for the top-level page sections. The section scaffold is static HTML, so this is the biggest piece (each section becomes a data-driven block).
- **Import** — paste a `TRIP` JSON to load a whole trip (needs the full-app re-render path; export already ships). The render functions are now complete enough to drive a full re-render — the remaining work is re-running them all + re-wiring non-delegated bits (calendar, maps/search) from a fresh `TRIP`.
- **Drag-reorder** — optional enhancement layered on the buttons. WCAG 2.2 SC 2.5.7 requires the single-pointer alternative regardless, so the buttons stay either way.

## Later (ideas)
- Multiple saved trips + a trip switcher.
- Share a trip via URL or a single JSON file.
- BYOK AI helper (bring-your-own Claude key) for "plan my week" / suggestions — deferred until there's a clean offline-friendly story.
- Theme/accent customization inside the editor.

Structural changes persist as a whole-array snapshot (`periplus.arrays.v1`; indices shift, so a field diff won't do) and survive reload; reset clears them. research(2026-05): chose move-buttons over a drag library — WCAG 2.2 SC 2.5.7 mandates a single-pointer alternative to dragging anyway, and buttons keep the app zero-dependency (drag is a future enhancement). These sections re-render safely (no per-card listeners to rebind). Covered by `tests/editing.spec.js`.
- [x] **No-code editing, Stage 3 — list & thin sections** — item-level add / delete / reorder *inside* the two-level sections (week plans, checklist items) and the thin timeline rows, completing card-and-item CRUD across the whole itinerary. The `LISTS` registry now resolves nested array paths (e.g. `weeks.0.items`) and snapshots the whole top-level array. The per-render wiring re-runs on re-render (current-week auto-open), and checkbox state moved to **one delegated `change` listener per stable list container** so it survives the re-render with no rebind. research(2026-05): event delegation over re-binding per checkbox — re-attaching listeners after a re-render is the anti-pattern (duplicate-firing); a single delegated listener on the persistent parent is the modern pattern (gomakethings / jasonformat). Open accordions stay open across a re-render; a new entry takes focus, a delete returns focus to a stable control (WCAG 2.4.3 focus order). Thin timeline rows place the controls inline at the row end via a 3-column edit-mode grid (the card corner placement doesn't fit a one-line row).
- [x] **Edit toggle reachable on mobile** — the topbar Edit toggle overflowed off the right edge at phone widths (four utility items in a no-wrap flex row → ~64px horizontal overflow on a 390px screen, toggle entirely off-screen), which had quietly made the entire editing feature desktop-only on a mobile-first PWA — uncaught because every e2e ran at desktop width. Fixed with `flex-wrap` so the actions wrap to a second row. research(2026-05): for a small button count, `flex-wrap` is the endorsed responsive-toolbar pattern (an overflow/hamburger menu is for many items). Now covered by an iPhone-13-width e2e (no sideways scroll + toggle within the viewport). Suite is 21 e2e total.
- [x] **No-code editing, Stage 4 — whole weeks & checklist groups** — top-level add / delete / reorder for the two-level sections: add a whole week or a whole checklist group (each seeded with one item) and reorder / remove them, completing structural CRUD *within* these sections. Driven by the same generalized handler (a top-level `blank` per `LISTS` entry; the section-level "+ Add week" / "+ Add group" button comes free from the inject-on-`blank` loop). research(2026-05): the per-week / per-group controls render **in the accordion body, never the `<summary>`** — a `<summary>` carries implicit button semantics, so interactive controls nested inside it are imperceptible to screen readers (Hassell Inclusion / Aditus / W3C APG); a labeled "Whole week" / "Whole group" strip keeps them distinct from the item controls. Covered by `tests/editing.spec.js` (23 e2e total). SW cache v6 → v7.
