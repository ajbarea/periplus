# Periplus Roadmap

Periplus is a fork-it-yourself, offline-first travel-itinerary PWA. **Edit mode** lets you change every field with zero code and add, remove, and reorder cards, list items, _and_ whole weeks / checklist groups — then **export or import** the whole trip as JSON to back it up or share it (saved on-device); the `TRIP` block in `app/index.html` remains the source for forking. The north star: full no-code authoring — add, remove, and reorder the top-level page sections too.

## Now (shipped)

- **Single source of truth** — every section and card renders from the `TRIP` object: directories, calendar, timelines, intel, checklists, venue, contacts. Edit one block, redeploy, it's your trip.
- **Tap-to-navigate** (Google/Apple Maps) and **add-to-calendar** (Google/Apple) on every relevant item.
- Installable, offline-first, prints clean.

## Next: no-code editing — remaining structure

Stages 1 (field editing), 2 (card sections), 3 (list & thin sections), 4 (whole weeks & groups), and 5 (JSON import) shipped (see Done). Remaining:

- **Sections** — add, delete, reorder, rename, and pick a type for the top-level page sections. The section scaffold is static HTML, so this is the biggest piece (each section becomes a data-driven block).
- **Full-trip portability** — trip dates (`TRIP_START`/`TRIP_END`) and the bespoke "live status" now-card copy aren't in the `TRIP` model yet, so import restores content but not the calendar's date range or the now-card text. Moving dates into `TRIP` (and making the now-card data-driven) would let import author an _arbitrary_ trip, not just restore this one's content.
- **Drag-reorder** — optional enhancement layered on the buttons. WCAG 2.2 SC 2.5.7 requires the single-pointer alternative regardless, so the buttons stay either way.

## Later (ideas)

- Multiple saved trips + a trip switcher.
- Share a trip via URL or a single JSON file.
- BYOK AI helper (bring-your-own Claude key) for "plan my week" / suggestions — deferred until there's a clean offline-friendly story.
- Theme/accent customization inside the editor.
- **Embedded offline maps — accepted (AJ, 2026-05-27); build after the NC trip (post-May 31).** Navigation today is tap-to-open Google/Apple Maps, which needs signal — exactly what fails in the dead zones the app targets. A true in-app offline map (single-file **PMTiles** vector tiles + **MapLibre GL**, no API key, OPFS-stored) closes that gap. This is a **knowingly-accepted exception to the zero-dependency / single-file ethos** (adds `maplibre-gl` + a multi-MB regional tile extract) — scoped deliberately, not a default, and not before the trip the app is currently loaded for. research(2026-05): docs.protomaps.com/pmtiles/maplibre, github.com/makinacorpus/maplibre-offline-pmtiles.

## Invariants

- **Internal docs live in `planning/`, never `docs/`.** Zensical publishes _everything_ under `docs/` onto the public site — a near-leak of the spec/plan was caught at ship time. Design notes, specs, and plans stay in `planning/` (outside the build).
- **Zero-dependency, no-build.** The app is a single `app/index.html` with all trip data in the `TRIP` block; forking = edit that block. Features stay vanilla (no-code editing shipped without adding a runtime dep) — weigh any library hard against this.
- **Service worker scoped to `/app/`** so it never hijacks docs-site navigations (Option-A topology: docs at the Pages root, PWA at `/app/`).

- [x] **No-code editing, Stage 2 — card sections** — add / delete / reorder entries across the card-grid sections (food, outdoor, intel, contacts, anchors) in edit mode via single-pointer buttons (↑ ↓ ×) + an "Add entry" button, driven by a small `LISTS` registry (grid + render + blank template per array). Structural changes persist as a whole-array snapshot (`periplus.arrays.v1`; indices shift, so a field diff won't do) and survive reload; reset clears them. research(2026-05): chose move-buttons over a drag library — WCAG 2.2 SC 2.5.7 mandates a single-pointer alternative to dragging anyway, and buttons keep the app zero-dependency (drag is a future enhancement). These sections re-render safely (no per-card listeners to rebind). Covered by `tests/editing.spec.js`.
- [x] **No-code editing, Stage 3 — list & thin sections** — item-level add / delete / reorder _inside_ the two-level sections (week plans, checklist items) and the thin timeline rows, completing card-and-item CRUD across the whole itinerary. The `LISTS` registry now resolves nested array paths (e.g. `weeks.0.items`) and snapshots the whole top-level array. The per-render wiring re-runs on re-render (current-week auto-open), and checkbox state moved to **one delegated `change` listener per stable list container** so it survives the re-render with no rebind. research(2026-05): event delegation over re-binding per checkbox — re-attaching listeners after a re-render is the anti-pattern (duplicate-firing); a single delegated listener on the persistent parent is the modern pattern (gomakethings / jasonformat). Open accordions stay open across a re-render; a new entry takes focus, a delete returns focus to a stable control (WCAG 2.4.3 focus order). Thin timeline rows place the controls inline at the row end via a 3-column edit-mode grid (the card corner placement doesn't fit a one-line row).
- [x] **Edit toggle reachable on mobile** — the topbar Edit toggle overflowed off the right edge at phone widths (four utility items in a no-wrap flex row → ~64px horizontal overflow on a 390px screen, toggle entirely off-screen), which had quietly made the entire editing feature desktop-only on a mobile-first PWA — uncaught because every e2e ran at desktop width. Fixed with `flex-wrap` so the actions wrap to a second row. research(2026-05): for a small button count, `flex-wrap` is the endorsed responsive-toolbar pattern (an overflow/hamburger menu is for many items). Now covered by an iPhone-13-width e2e (no sideways scroll + toggle within the viewport). Suite is 21 e2e total.
- [x] **No-code editing, Stage 4 — whole weeks & checklist groups** — top-level add / delete / reorder for the two-level sections: add a whole week or a whole checklist group (each seeded with one item) and reorder / remove them, completing structural CRUD _within_ these sections. Driven by the same generalized handler (a top-level `blank` per `LISTS` entry; the section-level "+ Add week" / "+ Add group" button comes free from the inject-on-`blank` loop). research(2026-05): the per-week / per-group controls render **in the accordion body, never the `<summary>`** — a `<summary>` carries implicit button semantics, so interactive controls nested inside it are imperceptible to screen readers (Hassell Inclusion / Aditus / W3C APG); a labeled "Whole week" / "Whole group" strip keeps them distinct from the item controls. Covered by `tests/editing.spec.js` (23 e2e total). SW cache v6 → v7.
- [x] **No-code editing, Stage 5 — JSON import (export ↔ import round-trip)** — paste a trip you exported and load it; it replaces the editable content on this device and persists (`periplus.trip.v1`), with field edits / array snapshots layering on top and reset clearing all three keys. Driven by a `renderAll()` that re-runs every content renderer in place — the calendar was extracted from its load-time IIFE into a re-callable `renderCalendar()`; the wiring (maps / search / share / calendar nav) sits on static elements and survives. research(2026-05): client-side `JSON.parse` in a try/catch + a light shape check, inline `role="alert"` error, no server — the standard offline-PWA import pattern. Scope is the portable `TRIP` content: trip dates and the bespoke "live status" now-card are deploy-side, not in the data model (see Next → Full-trip portability). Covered by `tests/editing.spec.js` (25 e2e total). SW cache v7 → v8.
